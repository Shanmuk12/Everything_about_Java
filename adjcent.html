01

Rows & Columns — What They Actually Mean

When you write arr[3][4], the first bracket is always rows, the second is always columns. A row is a complete horizontal strip; a column is a vertical slice. This is the same notation in both languages — but what those dimensions physically create in memory is completely different.

// A 3×4 grid — 3 rows, 4 columns
First bracket

Rows — [3]

How many horizontal strips exist. In Java this is the count of row objects that will be created on the heap.

arr[3][4] → 3 rows

Second bracket

Columns — [4]

How many cells wide each row is. In C this is the stride used in the offset formula to navigate flat memory.

arr[3][4] → 4 columns

02

Java Memory Model — Array of References

Java does not store a 2D grid. It stores an array of references (pointers), where each reference points to an independent row array living somewhere on the heap. When you write int[][] arr = new int[3][4], Java creates 4 separate objects: one primary reference array with 3 slots, and three separate int[4] arrays.

// Java heap after: int[][] arr = new int[3][4]
🧠
The key insightJava's 2D array is not a grid. It's an array whose slots hold the addresses of other arrays. Those inner arrays are fully independent objects. This is why each row can have a different length (jagged arrays), and why arr[r].length is a separate property per row.
03

C / C++ Memory Model — One Flat Block

C allocates a single contiguous block of integers. There are no row objects, no pointers, no indirection. The compiler uses the formula offset = row × COLS + col to find any element. What you call "rows" are just named regions inside one big flat line of memory.

// C: int matrix[3][4] — 12 consecutive integers in RAM
⚡
Why C is faster for matrix traversalWhen you loop row-by-row in C, you're reading adjacent memory locations. The CPU's prefetcher loads them ahead of time into L1 cache. In Java, each row is a separate heap object — they may be far apart in RAM, causing cache misses.
04

Declaration Rules — The Root Cause

Because the two languages have fundamentally different memory models, the dimension that must be provided at compile time is opposite in each language. This is not arbitrary — each language requires exactly the dimension it needs to do its first step of memory setup.

Java rule

LEFT dimension (rows) must be given

Java's first job is to build the primary reference array. To allocate it, Java must know how many reference slots to create. That count is the row count — the left bracket.

Once the primary array exists, column sizes can be filled in later (per row, even differently). That's why new int[3][] is valid.

Without the left dimension, Java can't even begin. new int[][3] fails instantly.

C / C++ rule

RIGHT dimension (cols) must be given

C's formula is offset = row × COLS + col. The compiler bakes this into every array access at compile time. Without COLS the formula is broken — the compiler can't generate any access code.

Row count can be inferred from the initializer or provided separately. int m[][3] is valid — the compiler counts rows from context.

Without the right dimension, the formula is undefined. int m[3][] fails.

08

Jagged Arrays — Java's Superpower, C's Impossibility

Because Java rows are independent objects, each row can have a different size. This is called a jagged (or ragged) array. C's flat-memory model makes true jagged arrays impossible — every row must be the same width for the formula to work.

Java — building a jagged array
// Step 1: build the primary array (row count must be known)
int[][] jagged = new int[3][];

// Step 2: each row gets its OWN size
jagged[0] = new int[2];   // row 0 has 2 elements
jagged[1] = new int[4];   // row 1 has 4 elements
jagged[2] = new int[1];   // row 2 has 1 element

// Traversal — use jagged[row].length per row!
for (int r = 0; r < jagged.length; r++) {
    for (int c = 0; c < jagged[r].length; c++) {  // ← per-row length!
        System.out.print(jagged[r][c] + " ");
    }
}
// Jagged array in memory — each row object is a different size

DSA Use Case — Adjacency List for Graphs

Java — graph adjacency list using jagged array
// A graph with 5 nodes. Node 0 connects to 2 others, node 1 to 4 others, etc.
int[][] adj = new int[5][];

adj[0] = new int[]{1, 2};            // node 0 has 2 neighbours
adj[1] = new int[]{0, 2, 3, 4};    // node 1 has 4 neighbours
adj[2] = new int[]{0, 1};            // node 2 has 2 neighbours

// Much more memory-efficient than a 5×5 grid (25 cells vs 8 cells used)
// This is how most graph problems are represented in competitive programming!

DSA Cheat Sheet

Java — Quick Reference

Java 2D array patterns
int[][] m = new int[3][4];   // 3 rows, 4 cols — standard
int[][] j = new int[3][];    // jagged — assign per row

m.length       // → 3  (number of rows)
m[0].length    // → 4  (number of cols in row 0)
m[1].length    // → 4  (same for uniform, can differ for jagged)

// Safe traversal template
for (int r = 0; r < m.length; r++) {
    for (int c = 0; c < m[r].length; c++) { // use m[r].length not m[0].length for jagged safety
        int val = m[r][c];
    }
}

// For-each (read-only)
for (int[] row : m)
    for (int val : row)
        System.out.println(val);

C / C++ — Quick Reference

C/C++ 2D array patterns
int m[3][4];              // 3 rows, 4 cols — stack allocated
int m[][4] = {{...}};    // cols required, rows inferred

// Size (only works for stack arrays, not pointers)
int rows = sizeof(m) / sizeof(m[0]);    // → 3
int cols = sizeof(m[0]) / sizeof(int); // → 4

// Traversal — row-by-row is cache-optimal
for (int r = 0; r < 3; r++)
    for (int c = 0; c < 4; c++)
        m[r][c];   // compiler sees: *(m + r*4 + c)
